Use this guide when you want to deploy a standard SvelteKit app with StackKits and need to decide between the standalone CLI path and the guided kombify-TechStack path.

<Note>
  Both paths use StackKits as the source of truth. The CLI is the direct operator path. kombify-TechStack is the orchestration layer with a SvelteKit UI, node inventory, job status, and agent-based execution.
</Note>

## Choose a rollout path

| Decision | CLI + one-liner | kombify-TechStack |
|----------|-----------------|-------------------|
| First user decision | Where does the one-liner run? | What outcome or use case do you want? |
| StackKit standard | You choose `base-kit`, `modern-homelab`, or `ha-kit`. | TechStack recommends the standard from intent and target evaluation. |
| User input style | Commands and spec values. | Use-case clicks, intent text, target selection, confirmation. |
| System help | Defaults and validation. | Pre-evaluation, recommendation, plan risk, post-deployment health. |
| Best fit | Technical operator path. | Guided user experience. |

## Required decisions

| CLI user must decide | TechStack user must decide |
|----------------------|----------------------------|
| Run the one-liner on the target server, on a local Docker host, or from a workstation with SSH access. | Select use cases such as app hosting, media, vault, files, smart home, or describe the desired homelab in plain language. |
| Choose the StackKit standard explicitly. Start with `base-kit` for the first SvelteKit app rollout. | Select or connect the target server/agent. |
| Enter technical values: domain/access mode, app image, port, hostname, health path, env, and secret references. | Provide only missing facts: domain preference, app artifact, secret source, or approval. |
| Review `stackkit plan` and run `stackkit apply`. | Review TechStack's recommendation and click apply. |

TechStack takes over the standard selection. It evaluates the intent, node resources, port/DNS/security fit, generated plan, route health, login path, and app health after rollout.

## Standard SvelteKit app contract

A SvelteKit app deployment needs the same core information in either path:

| Field | Purpose |
|-------|---------|
| App name | Stable service and route name. |
| Artifact | A container image such as `ghcr.io/acme/my-app:latest`, or a buildable source repository when your workflow supports it. |
| Port | The internal app port, commonly `3000` for the SvelteKit Node adapter. |
| Hostname | The final route, for example `app.home.lab`. |
| Health path | Prefer `/health`, following the kombify health contract. |
| Auth policy | Default to the StackKits login gateway unless the route is intentionally public. |
| Environment | Non-secret runtime values such as `PUBLIC_*`. |
| Secrets | Secret references only. Do not put literal secrets into the spec. |

Example intent:

```yaml stack-spec.yaml
stackkit: base-kit

network:
  domain: home.lab

apps:
  web:
    kind: sveltekit
    image: ghcr.io/acme/my-sveltekit-app:latest
    port: 3000
    route:
      host: app.home.lab
      auth: login-gateway
    health:
      path: /health
    env:
      PUBLIC_APP_NAME: "My App"
```

## Roll out with the CLI

<Steps>
  <Step title="Choose where the one-liner runs">
    Decide whether you run the installer on the target server, on a local Docker host, or on a workstation that can reach the target over SSH.
  </Step>

  <Step title="Install StackKits">
    Paste the one-line installer into that shell:

    ```bash
    curl -sSL https://raw.githubusercontent.com/kombifyio/stackKits/main/install.sh | bash
    ```
  </Step>

  <Step title="Choose the StackKit standard">
    Choose the technical standard yourself. For the first SvelteKit app rollout, start with BaseKit:

    ```bash
    mkdir my-sveltekit-stack
    cd my-sveltekit-stack
    stackkit init base-kit
    ```
  </Step>

  <Step title="Add your SvelteKit app">
    Add the app fields to `stack-spec.yaml`: image, port, route, health path, environment, and secret references.
  </Step>

  <Step title="Prepare, plan, and apply">
    Validate the host, generate the deployment assets, preview the change, and apply it:

    ```bash
    stackkit prepare
    stackkit generate
    stackkit plan
    stackkit apply
    ```
  </Step>

  <Step title="Check status">
    Confirm that the app container, route, and health check are working:

    ```bash
    stackkit status
    ```
  </Step>
</Steps>

## Roll out with kombify-TechStack

<Steps>
  <Step title="Open TechStack">
    Open the kombify-TechStack dashboard and create a new stack.
  </Step>

  <Step title="Describe the outcome">
    Select use cases or write the intent, for example "deploy my SvelteKit app with login and a working route".
  </Step>

  <Step title="Let TechStack evaluate">
    TechStack evaluates the intent, target resources, DNS/access model, security baseline, and plan risk. It recommends the StackKit standard and add-ons instead of forcing you to choose them.
  </Step>

  <Step title="Connect the target">
    Select an existing node or connect a worker/agent so TechStack can run prechecks and apply the StackKits deployment.
  </Step>

  <Step title="Provide missing facts">
    Add only the facts TechStack cannot infer: app image or repository, domain preference, and secret source.
  </Step>

  <Step title="Review and apply">
    Review the recommendation, apply it, and follow the job status until route, login, and health evaluations pass.
  </Step>
</Steps>

## Verify

After either rollout path, verify the same outcome:

```bash
curl -fsS http://app.home.lab/health
```

Expected result:

- The app route resolves through the StackKits gateway.
- The health endpoint returns success for deployment health checks.
- The app UI route is protected by the login gateway unless you explicitly configured it as public.
- `stackkit status` or the TechStack job view shows the app as healthy.

## Troubleshooting

<AccordionGroup>
  <Accordion title="The app container starts, but the route does not work">
    Check the hostname, domain, reverse proxy labels, and whether the app listens on the configured internal port.
  </Accordion>
  <Accordion title="The health check fails">
    Make sure your SvelteKit app exposes `/health` and returns a successful response without requiring a browser session.
  </Accordion>
  <Accordion title="Secrets are missing at runtime">
    Use secret references from your selected execution path. Do not paste raw secret values into public specs or documentation examples.
  </Accordion>
</AccordionGroup>
